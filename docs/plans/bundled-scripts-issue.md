# Bundled scripts for `skills/spyglass/scripts/`

## Context

The skill currently ships only maintainer scripts (`validate_skill.py`, `validate_all.sh`). These additions are **user-facing helpers** that encode patterns Claude-with-skill (and users without Claude) currently reconstruct from prose — with known failure modes. Prototype them here; migrate the generally-useful ones upstream into `LorenFrankLab/spyglass` once they prove out. See [docs/plans/skill-expansion-roadmap.md](skill-expansion-roadmap.md) for the broader roadmap.

## Scripts to add

### Environment & safety (no DB required)

- [x] **`scrub_dj_config.py`** — read `dj_local_conf.json` / `~/.datajoint_config.json` and print with `database.password` and other secrets masked. Encodes the scrubbed-read pattern SKILL.md mandates; kills the "oops leaked password into conversation history" risk. Highest-value single bundle. *Upstream candidate: `spyglass.settings` or `spyglass.utils.config`.*
- [ ] **`inspect_nwb_lite.py`** — sibling safety rail to `scrub_dj_config.py`: where `scrub_dj_config` prevents secrets from entering Claude's context, this prevents a 2 GB NWB file from doing the same. Opens an NWB via `NWBHDF5IO` in lazy mode (or walks the backing h5py tree directly) and emits a **text-only schema summary**: neurodata object IDs, table names, per-table row counts, per-array dtype + shape (no values), subject + session metadata, extension namespaces used. Never reads array payloads — a 50-line summary instead of loading gigabytes. Runs on any NWB under `$SPYGLASS_RAW_DIR` without touching the DB. *Scope note:* complements (doesn't duplicate) `nwbinspector`, which validates NWB correctness; this is a *schema preview* for planning `insert_sessions` / `populate` against unfamiliar data. *Upstream candidate: `spyglass.data_import.summarize_nwb(path)` or a `python -m spyglass.data_import summarize` CLI.*
- [x] **`verify_spyglass_env.py`** — single sanity check that validates (1) `SPYGLASS_BASE_DIR` and subdirectories exist and are writable, (2) DataJoint can connect to the configured host, (3) critical pins (SpikeInterface, DataJoint, PyNWB) match Spyglass's expectations. Agents can run this at session start to detect drift before it becomes a silent `populate` failure. *Upstream candidate: `spyglass.utils.diagnostics`.*
- [ ] **`map_si_to_spyglass.py`** — check the installed SpikeInterface version and output the correct class/method names for that version (e.g., `SortingAnalyzer` vs. the older `WaveformExtractor` across the SI 0.100 boundary). Prevents agents from pasting stale SI patterns found in training data. *Implementation note:* the mapping is really data (version → API), so prototype as a tiny script over a structured YAML under `references/catalog/spikeinterface_api.yaml` rather than hard-coded Python. *Upstream candidate: `spyglass.spikesorting.utils.si_compat`.*
- [ ] **`cardinality_check.py`** — take a restriction, print `len()`, sample 3 rows, and name the PK fields still needed to narrow to one row. Small, teaching-oriented; gates Common Mistake #2 (too-loose restriction + `fetch1()`). *May collapse into a code block in [feedback_loops.md](../../skills/spyglass/references/feedback_loops.md) instead of a standalone script — decide during prototyping.*

### Pattern helpers (structural footgun fixes)

- [ ] **`fetch_merge_row.py`** — given `(Master, key)`, do the merge-resolution chain — `merge_get_part(key) → fetch1('KEY') → (Master & merge_key)` — and return the restricted relation so the caller chooses the fetch shape (`.fetch1_dataframe()` for pandas-friendly pipelines like position / LFP, `.fetch_nwb()` for spike-sort or decoding outputs backed by NWB objects, `.fetch1('KEY')` when you only need the key to feed downstream, plain `.fetch1()` for a dict of fields). The script's job is to retire Common Mistake #1 by getting the merge-resolution chain right; the fetch shape is not its business. CLI flag `--as {relation,key,dict,dataframe,nwb}` can offer a pass-through for common cases without foreclosing the others. *Upstream candidate: method on the `_Merge` mixin — likely `resolve_merge(key)` returning the restricted relation, alongside `merge_get_part` / `merge_restrict`.*
- [ ] **`safe_delete_preview.py`** — given a restriction, show what `cautious_delete` *would* remove (counts per affected table, orphan intervals, downstream merge rows) without calling `.delete()`. Turns the inspect-before-destroy pattern in [destructive_operations.md](../../skills/spyglass/references/destructive_operations.md) into runnable code that enforces Core Directive #1. *Upstream candidate: `.delete_preview()` / `cautious_delete(dry_run=True)` on `SpyglassMixin`.*
- [ ] **`trace_delete_blockers.py`** — complement to `safe_delete_preview`. When a `.delete()` call has *already* failed with `IntegrityError` (FK constraint) or `PermissionError` (team-based check), walk the child graph and return the concrete blocker list: "row in `CurationV1` (merge_id=…) owned by `jsmith` is blocking". Retires the manual `parents()`/`children()` walk users currently do by hand. *Symmetry note:* Spyglass already ships `HelperMixin.find_insert_fail()` for the INSERT-side of the same problem (see "Thin wrappers" below). This script adds the DELETE-side mirror. *Upstream candidate: `SpyglassMixin.diagnose_delete_failure()`.*
- [ ] **`validate_lfp_params.py`** — pre-flight check for a proposed `LFPSelection` / `LFPBandSelection` key. Validates the chosen `FirFilterParameters` against `target_sampling_rate` (Nyquist: `target_sampling_rate ≥ 2 × high_cutoff`), filter order vs. transition band, and other silent-aliasing hazards. Structural fix for the single most common "silent wrong data" bug in the LFP pipeline. *Scope note:* this is the first **pipeline-specific** pre-flight in the bundle — if it proves its weight, expect siblings for spike-sort / DLC / decoding, each keyed to hazards documented in the roadmap's `catalog/quality_metrics.yaml`. *Upstream candidate: `LFPSelection.validate(key)` / `FirFilterParameters.check_compatible(lfp_key)`.*
- [ ] **`check_merge_integrity.py`** — pre-flight check for merge-table operations. For a given merge master (e.g. `SpikeSortingOutput`, `LFPOutput`, `PositionOutput`, `LinearizedPositionOutput`, `DecodingOutput`), verify that every `merge_id` in the master has exactly one corresponding part-table row, and every part-table row is reachable from the master. Catches orphans from interrupted `populate()` calls and manual deletes that bypassed `cautious_delete`'s cascade. Retires Common Mistakes #1 and #6 in their "the registry itself is already broken" shape — `merge_get_part` failures that look like bugs in your query but are actually registry drift. *Upstream candidate: method on `_Merge` mixin — `MergeMaster.check_integrity()` alongside `merge_restrict` / `merge_get_part`.*

### Data exploration, debugging & workflow (DB access required)

- [ ] **`session_summary.py`** — given an `nwb_file_name`, report which pipelines have results (position, LFP, spike-sort, decoding, ripple, linearized position) by walking the five merge masters. Every user runs this in their first hour; implements the "Preview Analysis Results" block of [workflows.md](../../skills/spyglass/references/workflows.md). *Upstream candidate: `spyglass.common.session_overview(nwb_file)`.*
- [ ] **`pipeline_provenance.py`** — given a `merge_id`, walk upstream and print the full parameter / selection / computation chain. Foundation for reproducibility and cross-dataset meta-analysis (roadmap Aims 1 + 3). *Upstream candidate: method on `SpyglassMixin`; complements the existing export serialization.*
- [ ] **`diff_keys.py`** — given two key lists ("populated successfully" vs. "failed"), return the differentiating column(s). Collapses the "why did populate fail on these 5 but not these 47" debugging question into one call. *Upstream candidate: `spyglass.utils.debug.diff_keys`.*
- [ ] **`fetch_figurl_labels.py`** — given a FigURL / sortingview URI and a curation `merge_id`, call the kachery-cloud API to pull the latest manual curation labels and emit a DataJoint-ready insert dict for the curation table. Closes the UI-back-to-DB toil loop in the spike-sort flow that [figurl.md](../../skills/spyglass/references/figurl.md) currently leaves manual. *Scope note:* this one does write to the DB — it's a standard selection-insert into the curation table, so it stays within the "normal selection-insert flow" carve-out in the Out-of-scope list below; it is *not* a delete/drop/overwrite. *Upstream candidate: method on `CurationV1` / `MetricCuration` — e.g., `CurationV1.sync_from_figurl(uri, merge_id)`.*
- [ ] **`reproduce_populate_failure.py`** — after a parallel `populate(reserve_jobs=True, suppress_errors=True)` run, DataJoint's `~jobs` table logs errors per failed key. This script pulls the most recent error rows for a given table, extracts the exact primary key that failed, and prints a copy-pasteable snippet to reproduce it: `<TableClass>().populate(<key_dict>)` with `reserve_jobs=False, suppress_errors=False` so the traceback surfaces directly. Eliminates the "triage loop" where the agent re-walks logs and grep output to isolate which key failed. Verified Spyglass doesn't wrap the jobs table (zero matches for `fetch_jobs` / `reserve_jobs` helpers in source), so this is filling a real gap rather than re-skinning existing capability. *Upstream candidate: `spyglass.utils.debug.reproduce_populate_failure` or a `PopulateMixin.replay_last_failure()` method.*
- [ ] **`check_schema_integrity.py`** — compare the DB's actual MySQL schema against the Python class `.definition` strings and report drift: columns added to Python that aren't in the DB, columns in the DB that Python no longer declares, type changes. Critical symptom it solves: after `git pull`, someone merged a schema change without calling `.alter()`, and your next populate raises `KeyError` or silent `DataError`. DataJoint validates class-to-DB on instantiation but doesn't diff — this script does. Proactively suggests the `.alter()` call before runtime failure. *Upstream candidate: this arguably belongs in DataJoint itself (Python ↔ DB definition diff is framework-general), but Spyglass conventions like `SpyglassMixin` / merge-table shape may want lab-specific checks layered on top. Start as a Spyglass-side utility and upstream to `dj.schema.check_integrity()` if the pattern generalizes.*
- [ ] **`dandi_preflight.py`** — catches missing DANDI metadata *before* hours-long uploads fail. For a given `merge_id` or analysis-NWB path, (a) validate the NWB file with `dandi.validate.validate` (already in Spyglass's transitive deps via `common_dandi.py`), and (b) cross-reference `Subject` + `Session.Experimenter` + `Session` rows in DataJoint to confirm DANDI-required fields are populated (species, sex, age, lab, experimenter, institution, session_description). Reports specific missing fields with pointers to the table that needs updating. *Scope note:* complements the existing [`validate_dandiset(folder)`](../../skills/spyglass/references/export.md) at `common_dandi.py:357`, which runs *post-export* on the assembled folder. This one runs *pre-export* on individual files + DataJoint state — catches "Subject row missing `species`" months before it blocks a DANDI upload. *Note on nwbinspector*: the proposal named it, but nwbinspector is **not** a Spyglass dep. Use `dandi.validate` to match the existing surface, or add nwbinspector explicitly if NWB-schema-compliance checks (not DANDI-compliance) become valuable. *Upstream candidate: `spyglass.common.common_dandi.preflight(key)` next to `validate_dandiset`.*
- [ ] **`visualize_schema_neighborhood.py`** — given a table name, emit a text / Mermaid / DOT representation of its immediate parents and children (1-hop neighborhood). Agent-facing output is the win: instead of Claude calling `.parents()` + `.children()` + `.describe()` repeatedly to understand a new table, it gets "Table A depends on B and C; D and E depend on A" in one call, with Mermaid available for an LLM-readable text format. Different from `trace_restriction.py` (Thin wrappers section), which does full cascade walks for restriction propagation — this one is scoped to the 1-hop local map. *Upstream candidate: method on `SpyglassMixin` — `MyTable.show_neighborhood(format='mermaid')` alongside `parents()` / `children()`.*
- [ ] **`disk_usage_report.py`** — lab-wide aggregator over `HelperMixin.get_table_storage_usage` (reference-only today) sliced by experimenter / subject / pipeline. Output is a cross-table report: "rows J16/alice/SpikeSort=38 GiB; J20/bob/LFP=10 GiB; ..." for all interesting groupings. Addresses [#1359 "Disk space management for rec files"](https://github.com/LorenFrankLab/spyglass/issues/1359). Genuinely new capability over `get_table_storage_usage` (per-table bytes) — this is the cross-cutting aggregation that turns raw byte counts into data-management decisions (which experimenters' raw data is archivable; which pipeline outputs are cheap to regenerate vs. worth keeping). Flags: `--groupby experimenter,subject,pipeline`, `--min-gb N` to filter small rows, `--csv` for spreadsheet-friendly output. *Upstream candidate:* `spyglass.utils.storage.disk_usage_report(groupby=...)` next to `HelperMixin.get_table_storage_usage`.
- [ ] **`describe_params.py`** — closes the single biggest hallucination vector in the skill: LLMs can't see `*Params` Lookup contents and guess names from training data. Invocation: `python describe_params.py --pipeline SpikeSortingV1 --json` lists every `*Params` row feeding the pipeline (`SpikeSorterParameters`, `SpikeSortingPreprocessingParameters`, `ArtifactDetectionParameters`, `WaveformParameters`, `MetricParameters`) with: params name, contents blob, usage count (how many Selection rows reference it), last-updated timestamp if available. The LLM now has ground truth instead of guessing `"default"` and hitting `DataJointError: row not found`. Accepts `--table <TableName>` for a single Params table. Surfaces drift instantly — if the blob contains a deprecated field name (e.g. [`local_radius_um` before the `radius_um` rename](https://github.com/LorenFrankLab/spyglass/issues/1324)), it's visible in the output. *Upstream candidate:* `spyglass.utils.params.describe(pipeline)` — sibling to `HelperMixin.get_params_blob_from_key`.

### Thin wrappers over existing Spyglass utilities (don't reinvent)

Spyglass already ships a handful of debugging helpers that are poorly surfaced — no tutorial mentions, buried import paths, and output that needs reshaping for terminal consumption. These bundled scripts are *thin wrappers*: they shell out to the upstream method and pretty-print the result. Lowest-effort, highest-clarity additions.

- [ ] **`check_analysis_files.py`** — wraps `spyglass.common.AnalysisNwbfile.check_all_files()` (backed by `common_file_tracking.AnalysisFileIssues`). Scans every registered analysis table for missing / unreadable / checksum-mismatched files, and with `--resolve` reports which downstream tables are blocked by each corrupted input. Subsumes what we'd otherwise prototype as `audit_analysis_nwbfiles.py`. Script job: call the method, group results by affected table, exit non-zero if any issues found (so CI / pre-populate hooks can gate on it). *Upstream: already there — our ask is just discoverability + a CLI face.*
- [ ] **`trace_restriction.py`** — wraps `spyglass.utils.dj_graph.RestrGraph` (and `TableChain` for shortest-path). Given a seed table + restriction + direction (`up` / `down`), cascade through the dependency DAG and emit the full list of ancestor / descendant tables and row counts touched. Enables "what does this restriction actually affect across my whole pipeline?" answers that currently require manual graph walks. Script job: convert the graph object into a human-readable tree. *Upstream: already there; our ask is a usable front door.*
- [ ] **`diff_nwb.py`** — wraps `spyglass.utils.recompute_helper_fn.H5pyComparator(old, new)`. Given two analysis-NWB paths (or h5py objects), walk both trees and print a structured diff: mismatched datasets, missing groups, divergent scalar values. Use case: "I re-ran this analysis and it produced a subtly different NWB — what changed?" The upstream class is even tagged `# pragma: no cover # informational, not functional` — the maintainers explicitly treat it as a diagnostic. Script job: accept two paths on CLI, open both as h5py, pass to `H5pyComparator`, exit non-zero if differences found (so it can gate reproducibility checks in CI). Complements `pipeline_provenance.py`: provenance explains *how* an analysis was produced, `diff_nwb` explains *what actually came out differently*. *Upstream candidate: `spyglass.utils.diff_nwb` as a first-class `python -m` CLI.*

## Design conventions

These cross-cutting requirements apply to **every** script on this list, not just new ones. The goal is: an LLM running the skill should be able to cite any script's output as *evidence*, not as *narrative*, and a future-reviewer looking at a captured log file should be able to trace exactly which code produced it.

### 1. `--json` mode with a reproducibility envelope

Every script that produces decision-grade output ships a `--json` flag. The JSON payload is wrapped in a small envelope carrying environment fingerprinting, so the output is self-contained evidence:

```json
{
  "script": "verify_spyglass_env.py",
  "script_version": "0.1",
  "spyglass_version": "0.5.5",
  "spyglass_commit": "a7ddbd3",
  "datajoint_version": "0.14.5",
  "python_version": "3.11.7",
  "timestamp_utc": "2026-04-22T14:10:00Z",
  "dj_user": "alice",
  "dj_host": "db.example.test",
  "results": { ... }
}
```

Keys without runtime access (e.g. `spyglass_commit` when Spyglass isn't importable) are set to `null`, never omitted — a consumer can rely on the shape.

Rationale: LLM claims like "your env is healthy" or "this analysis is reproducible" only hold if the JSON payload proves *which* env and *which* code produced the claim. Without the envelope, a captured log file is useless a month later.

**Retrofit plan.** Already-shipped scripts (`scrub_dj_config.py`, `verify_spyglass_env.py`) emit JSON without the envelope today. Just fix them — wrap the current `results` / scrubbed config inside the envelope, update tests to match. No external consumers yet, so the JSON-schema change is acceptable. Factor the envelope builder into `scripts/_envelope.py` so the third script onward doesn't reinvent it.

### 2. Bounded runtime

Scripts that touch the network (DB connection, kachery, DANDI) have a `--timeout` flag with a sane default (≤ 10 s). Daemon-thread pattern from `verify_spyglass_env.py:check_dj_connection` is the reference — a worker thread attempts the operation; the main loop joins with timeout and abandons the worker if it hangs. `signal.alarm` is not portable and doesn't interrupt socket hangs reliably.

### 3. Never-leak invariant for scripts that read sensitive state

Scripts that read DataJoint config, kachery credentials, or any file that may contain secrets MUST preserve the `scrub_dj_config.py`-style invariant: on *any* failure path, stdout is empty — the script never falls back to printing raw file bytes when scrub fails. Traceback messages (which may contain connection strings with credentials) are replaced with exception-class-name-only reporting. Reviewed and tested for `scrub_dj_config.py` and `verify_spyglass_env.py:check_dj_connection`; any new network-touching or secret-reading script inherits this bar.

### 4. Structured evidence over prose status lines

`CheckResult(name, status, message, evidence)` is the reference pattern (from `verify_spyglass_env.py`). The `evidence` dict is machine-parseable; the `message` is human-readable. Human and JSON modes render from the same data. LLMs grade on evidence; humans grade on message; neither is a second-class citizen.

### 5. Every script has a one-line upstream candidate

Module docstring includes a `Lifecycle:` paragraph naming the upstream migration target. Enforced by `scripts/README.md`'s script-adoption checklist. Prevents the prototype from becoming a permanent fork.

## Workflow

For each script:

1. **Prototype here** in `skills/spyglass/scripts/` with tests under `skills/spyglass/tests/`.
2. **Wire into the skill** — cite from the relevant reference; extend `allowed-tools` deliberately if Claude needs to run them directly (default: user runs, Claude reads).
3. **Propose upstream** via a draft PR against `LorenFrankLab/spyglass` once the script stabilizes.
4. **Retire locally** when upstream merges — update the skill's references to point at `spyglass.xxx.yyy` and remove the prototype. Any corresponding "Common Mistake" entry that the new API structurally prevents can be deleted from [common_mistakes.md](../../skills/spyglass/references/common_mistakes.md).

## Out of scope

- Anything that writes to the DB beyond Spyglass's normal selection-insert flow.
- Full workflow scripts (e.g., "ingest-then-populate-everything") — `populate_all_common` already owns that surface.
- Pipeline-specific canonical examples — those stay inline with their reference prose.
- **Re-implementing utilities Spyglass already ships.** When a helper exists upstream, bundle a thin wrapper (see the Thin wrappers section) rather than a clean-slate prototype. Checked during scoping: if a proposed script's behavior substantially overlaps with anything in `spyglass.utils.mixins.helpers.HelperMixin`, `spyglass.utils.dj_graph`, `spyglass.common.common_file_tracking`, or `spyglass.common.AnalysisNwbfile.check_all_files()`, demote it to a wrapper. One specific proposal that was deduplicated: a `verify_analysis_files.py` that walks `AnalysisNwbfile` rows and checks each file's on-disk existence — this IS `AnalysisNwbfile.check_all_files()` and is already covered by `check_analysis_files.py` in the Thin wrappers section.
- **Grep cookbooks as user-facing references.** A proposal to add `references/grep_patterns.md` (canonical source-search commands for "find all V1 pipelines", "find all SpyglassMixin subclasses", etc.) was considered and rejected. The skill routes by *topic / decision* — meta-tooling for searching the Spyglass repo is a Spyglass-developer concern, not a Spyglass-user one, and grep patterns drift faster than prose (any rename silently breaks them). If Claude is observed reconstructing the same greps across sessions, bundle `scripts/sp_grep.py --pattern <name>` where the patterns live in runnable Python that can be validated and tested — don't encode them as a static reference file.
